# What we learned

**Purpose:** Short, concrete entries so we do not repeat the same mistakes. Format: **symptom → cause → fix → rule for next time.**

_Add entries after non-obvious bugs or integration wins._

## Entries

_(Example:)_  
- **Symptom:** API returned 401 on demo. **Cause:** token in `.env.local` not loaded in production build. **Fix:** documented `VITE_*` prefix / build-time env. **Rule:** verify env in the same mode you demo (dev vs prod build).
- **Symptom:** Tab-switch detection only fires when leaving/returning to Zoned tab, not between other tabs. **Cause:** Browser security — Page Visibility API (`visibilitychange`) only reports changes for the *current* document; it cannot observe switches between other tabs. **Fix:** Added `window.blur`/`focus` for marginal improvement + implemented Document Picture-in-Picture so the session monitor floats on top while the user works in other tabs. **Rule:** Web pages cannot observe other-tab activity; for cross-tab monitoring, use PiP/popup to keep the monitor visible, or build a browser extension.
- **Symptom:** User marked "distracted" whenever they leave the Zoned tab, even for productive work (Gmail, Google Docs, PowerPoint). **Cause:** `isTabAway` was used as the distraction flag — any tab switch = distracted. **Fix:** Built a Chrome extension (MV3) that checks the destination tab's URL against 80+ distracting domains. Introduced `isDistractedByTab` (only true for distracting sites) vs `isTabAway` (true for any tab switch). Session page uses `isDistractedByTab` for the distraction indicator. Without the extension, tab-away is NOT counted as distraction (benefit of the doubt). **Rule:** Separate "away from our tab" from "on a distracting site" — they are fundamentally different signals.
- **Symptom:** Extension `importScripts('distraction-sites.js')` failed in MV3 service worker. **Cause:** MV3 service workers are more restrictive with `importScripts`. **Fix:** Switched to ES modules — added `"type": "module"` in manifest.json, used `export`/`import` syntax. **Rule:** Always use ES modules (`type: "module"`) for MV3 service workers; `importScripts` is legacy and fragile.
- **Symptom:** Extension alerts fired even when user was on dashboard/settings (no active session). **Cause:** `isZonedSessionOpen()` only checked if any `localhost:3000` tab was open, not whether a session was running. **Fix:** Added bidirectional messaging (page → content script → background) with `chrome.storage.local` to track actual session state. Added safety net clearing flag when all Zoned tabs close. **Rule:** Don't infer app state from tab presence; use explicit session lifecycle signals.
- **Symptom:** React 19 / Next.js 16 lint errors for `setState` in effects and `Date.now()` in render. **Cause:** Strict new lint rules (`react-hooks/set-state-in-effect`, `react-hooks/purity`) flag common patterns. **Fix:** Used `eslint-disable` blocks for intentional state resets in effects; moved `Date.now()` calls into interval callbacks. **Rule:** In React 19 projects, avoid synchronous setState at the top level of effects; compute time-dependent values inside callbacks.
- **Symptom:** Gaze detection was unreliable — head turns didn't register, looking up was ignored, AFK recovery was broken, and extreme rotation was needed. **Cause:** Five compounding issues: (1) no eye visibility check so occluded iris landmarks were trusted, (2) vertical away logic used `smoothY > threshold` instead of `Math.abs(smoothY)` — up was excluded, (3) head baseline was set once and never adapted, (4) calibration accepted any frame regardless of quality, (5) smoothing window of 8 was too aggressive. **Fix:** Added frame quality scoring (EAR + head angle), symmetric vertical detection, AFK recalibration, rolling head baseline via EMA, adaptive smoothing, head-only away detection for extreme angles, and low-quality grace period. **Rule:** Webcam gaze detection needs quality gating on every frame — never trust iris landmarks when eyes aren't demonstrably visible; always use `Math.abs()` for symmetric detection; baselines must adapt to posture changes.
- **Symptom:** Accountability partner never notified despite Twilio keys being correct. **Cause:** Four compounding bugs: (1) `accountabilityTriggeredRef` was a boolean — one trigger of any type permanently blocked ALL other triggers for the session; (2) the flag was set to `true` BEFORE the API call, so a network/Twilio failure silently locked out retries forever; (3) thresholds were wrong (30 min AFK / 5 gazes instead of spec's 15 min / 10 gazes); (4) no user feedback — `catch {}` silently swallowed errors with no toast. **Fix:** Changed ref to `Set<string>` for per-reason tracking, set flag only after `res.ok`, added toast success/error messages, corrected thresholds, added Twilio env validation at route level with stage-specific error responses. **Rule:** Never set success flags before the operation succeeds; use per-reason tracking for multi-trigger systems; always surface API errors to the user.
- **Symptom:** After AFK, system showed "Look at your screen to dismiss" but wouldn't resume — user stuck staring at a full-screen alarm. **Cause:** Three-part circular deadlock: (1) AFK set `isLookingAway=true` in React state but calibration path used `setState((p) => ({ ...p }))` which preserved that stale `true` — never reset it; (2) head baseline was kept from before AFK, so posture change caused large headYawDelta → `lookingAway=true` in gaze computation too; (3) head baseline EMA correction required `!lookingAway` which was stuck true because of the stale baseline. Session page used `gazeAway = isLookingAway || noFaceDetected` so even after face returned, alarm stayed up during recalibration. **Fix:** (a) Clear both gaze AND head baselines on AFK return. (b) All calibration setState calls now explicitly set `isLookingAway: false`, `isCalibrated: false` — no `...p` spread for critical fields. (c) Session page gates gaze distraction on `isCalibrated`: `gazeAway = noFaceDetected || (isCalibrated && isLookingAway)`. (d) Added `trackingPhase` (`lost` / `reacquiring` / `calibrating` / `active`) for explicit lifecycle. **Rule:** Never gate resume on a condition that itself requires the resumed system to be running. Separate "face is present" from "gaze is classified" — face presence gates recalibration, gaze classification gates distraction. Always clear stale baselines on context change; EMA drift correction can't fix large jumps.
